# Tagging Schema

**Project:** ha-web-portal · AWS us-east-1
**Applied:** April 2026

---

## Overview

A six-key tagging schema was designed and applied to all 16 taggable resources via AWS Tag Editor. Tags were activated as cost allocation dimensions in the Billing console, enabling Cost Explorer to group and filter costs by workload, environment, owner, and cost centre.

Tags applied at this point are forward-looking — they do not backfill historical Cost Explorer data. Historical costs are attributed using service-level proxy allocation.

---

## Tag Keys and Values

| Tag key | Value(s) | Scope |
|---|---|---|
| `Environment` | `production` | All resources |
| `Project` | `ha-web-portal` | All resources |
| `Owner` | `platform-team` | All resources |
| `CostCenter` | `CC-1001` | All resources |
| `Criticality` | `high` | All resources |
| `Workload` | `web-tier` · `data-tier` · `network` · `observability` | Varies by resource |

Tag keys are case-sensitive. Values must match exactly as specified — variations will not be recognised by Cost Explorer.

**Known gap — auto-generated resources**

Two CloudWatch alarms (`TargetTracking-ha-*`) were auto-generated by the ASG target tracking policy and arrived untagged. Manual remediation applied at inception. In production, AWS Config's `required-tags` rule surfaces such resources automatically within the weekly audit cycle.
